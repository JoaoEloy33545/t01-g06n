\subsection*{2.1 Coordenação de Transações Distribuídas — Modelo X/Open}
\addcontentsline{toc}{subsection}{2.1 Coordenação de Transações Distribuídas — Modelo X/Open}

\textit{(Discutir o problema de coordenação centralizada vs. distribuída de transações distribuídas. Apresentar o modelo X/Open DTP: papéis AP, TM e RM; interfaces TX (\texttt{tx\_begin}, \texttt{tx\_commit}, \texttt{tx\_rollback}) e XA (\texttt{xa\_prepare}, \texttt{xa\_commit}, \texttt{xa\_rollback}). Apresentar o protocolo Two-Phase Commit (2PC) e o Two-Phase Lock Manager (TPLM/2PL) como mecanismos de coordenação da concorrência. Mapear estes papéis para os elementos serviço implementados.)}

O modelo \textit{X/Open Distributed Transaction Processing} (DTP) define uma arquitetura
padronizada para a coordenação de transações em sistemas distribuídos, assente em três
componentes com responsabilidades bem delimitadas:

\begin{itemize}
  \item \textbf{Application Program (AP)} --- inicia e termina transações via interface TX;
        no sistema implementado, corresponde ao \texttt{Service.Client} e ao \texttt{Service.Validator}.
  \item \textbf{Transaction Manager (TM)} --- coordena o protocolo 2PC entre os RMs participantes;
        implementado como \texttt{Service.TM}.
  \item \textbf{Resource Manager (RM)} --- gere um recurso transacional e expõe a interface XA;
        implementado como \texttt{Service.Vector}.
\end{itemize}

\textit{(Detalhar o protocolo 2PC: Fase 1 Prepare — o TM envia \texttt{xa\_prepare} a todos os RMs e aguarda votos; Fase 2 Commit/Abort — conforme os votos, o TM envia \texttt{xa\_commit} ou \texttt{xa\_rollback}. Discutir o problema do TM como ponto de falha único e a necessidade de log persistente para recuperação.)}

\textit{(Detalhar o protocolo 2PL: fase de expansão — aquisição de locks antes de aceder aos dados; fase de contração — libertação após commit/rollback. Discutir a garantia de serializabilidade. Apresentar como exemplo concreto a verificação transacional do invariante: o \texttt{Service.Validator} adquire read locks em todos os elementos antes de somar, garantindo uma leitura consistente em execução concorrente.)}
